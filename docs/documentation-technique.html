Documentation technique

1.Vue d'ensemble

M3A Fleet est une application SaaS multi-tenant de gestion de flottes VTC. Un déploiement unique sert tous les clients (tenants) ; chaque tenant est isolé au niveau base de données, API et stockage de fichiers.

4.Sécurité

4.1 Isolation multi-tenant (défense en profondeur)

  • Niveau 1 — RLS PostgreSQL : policies par tenant_id sur toutes les tables métier (clé anon).
  • Niveau 2 — API : chaque route dérive le tenantId de la session serveur (requireAdminAuth() / requireAnyAuth()) — jamais du corps de la requête.
  • Niveau 3 — Storage : chemins forcés sous le préfixe du tenant, bucket privé, URLs signées 30 min.

4.2 Rôles et authentification

RôleAccèsMécanisme
Chauffeur (driver)Ses rapports, avances, documents, notificationsSession Supabase + RLS + requireAnyAuth
Admin (gestionnaire)Tout son tenant : validation, paiements, pilotage, exportsSession Supabase + rôle admin + requireAdminAuth
Superadmin (M3A)Tenants, plans, validation paiements, injection importsClé secrète server-only, comparaison constant-time, rate limit 5/15 min/IP

4.3 Mesures complémentaires

  • Upload : validation MIME par magic bytes, taille max 10 MB, path sanitisé anti-traversal ;
  • Inscription : rate limit 3 créations/h/IP, rollback transactionnel (tenant + auth + profil) ;
  • SUPABASE_SERVICE_ROLE_KEY et SUPERADMIN_KEY strictement server-side (jamais NEXT_PUBLIC_) ;
  • Middleware : vérification d'expiration du plan à chaque requête → redirection /locked ;
  • Journal audit_logs des actions sensibles.
Limite connue : les rate limits sont en mémoire (réinitialisés à chaque déploiement, non partagés entre instances Vercel). Migration prévue vers un store partagé (Upstash Redis) — voir roadmap.

6.Environnements & configuration

# Variables d'environnement (Vercel + .env.local — jamais commitées)
NEXT_PUBLIC_SUPABASE_URL=          # URL du projet Supabase
NEXT_PUBLIC_SUPABASE_ANON_KEY=     # clé anon (RLS appliquée)
SUPABASE_SERVICE_ROLE_KEY=         # SERVER ONLY — bypass RLS
NEXT_PUBLIC_DEFAULT_TENANT_SLUG=   # tenant par défaut en localhost
SUPERADMIN_KEY=                    # SERVER ONLY — console superadmin
NEXT_PUBLIC_SENTRY_DSN=            # monitoring
SENTRY_ORG= / SENTRY_PROJECT= / SENTRY_AUTH_TOKEN=
NEXT_PUBLIC_VAPID_PUBLIC_KEY= / VAPID_PRIVATE_KEY= / VAPID_SUBJECT=
  • Production : Vercel, branche main, déploiement automatique à chaque push.
  • Développement : npm run dev sur localhost:3000 avec .env.local.
  • Tenants : résolution par sous-domaine {slug}.fleet.m3asolutions.com (ou slug par défaut en local).

7.Qualité, CI/CD & exploitation

  • CI GitHub Actions (.github/workflows/ci.yml) : type-check tsc, ESLint, build production, npm audit — sur push et PR ;
  • Tests : Jest — moteur de rémunération et logique de plans (40 tests) ; npm run test:ci ;
  • Monitoring : Sentry (erreurs + sourcemaps) ; page publique /status ;
  • Sauvegardes : backups Supabase quotidiens (à vérifier selon le plan du projet) + export SQL des migrations versionné dans le repo ;
  • Procédures : voir le document SOP — Procédures opérationnelles (déploiement, incident, restauration, rotation des secrets).
